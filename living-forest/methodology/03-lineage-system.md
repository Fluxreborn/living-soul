# 03 - 血统系统 / Lineage System

## 为什么需要血统？

项目不是凭空产生的：
- **Soul Memory** 是从 **Token Saver** 的某个分支长出来的
- **Newsletter v2** 可能继承了 **Blog** 的内容策略
- 理解「从哪来」才能理解「为什么是现在这样」

## 血统的三个文件

### 1. 子项目的 _lineage.md

记录「我从哪来」：

```markdown
# 项目血统 / Project Lineage

## 出生记录
- **父项目**：`token-saver/`
- **父分支**：Token Saver / 004-SoulOS-萌芽
- **出生日期**：2026-03-23
- **接生员**：Fable + Karo

## 出生原因
TOKEN Saver 始终只是「防御性补丁」：
- 优化 token 消耗，但不改变本质
- 子 Agent 隔离，但增加复杂度

Soul Memory 是「根本性探索」：
- 重新定义 AI 的记忆方式
- 用「睡眠+抽签」替代「机械存档」

当讨论深入到「睡眠是否是升维归纳」时，
项目质地发生了质变，需要独立演化。

## 家族关系
```tree
Token Saver [父/归档]
├── 001-立项
├── 002-KPI 计量
├── 003-子 Agent 方案
└── 004-SoulOS-萌芽 ──┐
    ├── 胎盘：归档的 Token Saver SPEC
    └── 分娩 → Soul Memory [子/活跃]

Soul Memory [独立]
├── 000-血统（本文件）
├── 001-立项（从「睡眠」概念开始）
├── 002-抽签机
├── 003-亮度系统
└── ...
```

## 脐带保留（跨项目引用）
- 父项目的教训：`../token-saver/graveyard/003-agent-isolation.md`
  - 为什么废弃：复杂度高于收益
  - Soul Memory 的启示：简单机制（抽签）优于复杂隔离
- 共享组件：无（完全独立，无代码依赖）

## 独立演化追踪
| 日期 | Soul Memory 里程碑 | 与父项目的关系 |
|------|-------------------|---------------|
| 2026-03-23 | 立项/分娩 | 从 Token Saver 独立 |
| 2026-03-23 | 首次睡眠/抽签 | 无直接关联，完全自主 |

## 分娩的判断标准

什么时候该让分支独立成新项目？

| 情况 | 留在原项目 | 独立成新项目 |
|------|-----------|-------------|
| **目标变化** | 优化原目标 | 追求新目标 |
| **方法论变化** | 改进原方法 | 完全不同的方法 |
| **资源需求** | 共享资源 | 需要独立资源 |
| **团队变化** | 原团队维护 | 新团队接手 |
| **时间线** | 与原项目同步 | 独立时间线 |

**Soul Memory 的例子**：
- Token Saver 目标：减少 token 消耗
- Soul Memory 目标：重新定义 AI 记忆方式
- → 目标完全不同 → 必须独立

## 复活与血统

Graveyard 中的分支复活时：

**同项目内复活**：
```markdown
## 复活记录
- **2026-03-25**：006-向量数据库迁移 → 从 005-old 复活
  - 原 005-old「回声网络」废弃原因：技术不成熟
  - 复活条件已满足：向量数据库技术就绪
  - 新编号 006：表示这是新分支，不是原分支的简单恢复
```

**跨项目复活**（罕见）：
```markdown
## 跨项目复活
- **2026-04-01**：token-saver/graveyard/003-agent-isolation
  - 复活为独立项目：`agent-isolation-v2/`
  - 原因：Soul Memory 成功后，发现隔离方案可以反哺 KPI 计量
  - 血统：父 Token Saver，但独立演化
```
