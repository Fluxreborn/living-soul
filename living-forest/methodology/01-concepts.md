# 01 - 核心概念 / Core Concepts

## 三个记忆层级

人的记忆有层级，AI 的项目管理也应该有：

```
短期记忆（分钟-小时）
├── ACTIVE.md —— 当前在做什么
└── Session 上下文 —— 本轮对话内容

中期记忆（天-周）
├── _tree.md —— 项目结构（活树）
├── _forest.md —— 项目关系（森林）
└── trunk/branches/ —— 具体内容

长期记忆（月-年）
├── Soul Memory —— 人格化的经验签
├── MEMORY.md —— 人工维护的精华
└── graveyard/ —— 可复活的历史
```

## 活树的四个区域

### 1. Trunk（主干）
当前项目的核心演进线：
- 必须是**已完成**或**进行中**的状态
- 编号连续（001, 002, 003...）
- 每个节点有明确的完成标准

### 2. Branches（分支）
从主干延伸的实验性功能：
- 可能来自主干，也可能独立生长
- 状态可以是「研究中」「原型中」「待验证」
- 验证成功后合并回主干，或成为独立项目

### 3. Graveyard（墓地）
废弃的分支，但不删除：
- 记录废弃原因
- 标记复活条件
- 保留完整内容，随时可唤醒

### 4. Snapshots（快照）
树形结构的历史版本：
- `_tree.v1.md`, `_tree.v2.md`...
- 记录项目的演化轨迹
- 方便「考古」——查看某天的完整结构

## 森林的两张图

### 地图（Map）
当前所有活跃项目的结构：
```forest
AI 助手基础设施
├── OpenClaw 优化 [活跃]
└── Soul Memory [活跃]

内容生产
├── Newsletter [待命]
└── Blog [归档]
```

### 谱系（Genealogy）
项目间的分娩关系：
| 子项目 | 父项目 | 分娩原因 |
|--------|--------|---------|
| Soul Memory | Token Saver | 从防御补丁到根本探索 |

## 血统的三个要素

当一个项目从另一个项目「分娩」出来：

1. **接生记录**（Lineage Document）
   - 父项目 / 父分支
   - 分娩日期
   - 独立原因

2. **脐带保留**（Cross Reference）
   - 父项目标记「已长出子项目」
   - 子项目标记「从父项目分娩」
   - 双向可追踪

3. **独立演化**（Evolution）
   - 子项目有自己的 _tree.md
   - 不再依赖父项目的更新
   - 但保留血统查询能力

## 记忆加载策略

### 理想状态
```
新 Session 启动
    ↓
读 SOUL.md + USER.md（人格）
    ↓
读 ACTIVE.md（当前焦点）
    ↓
读 _forest.md（30 秒全局）
    ↓
用户提到「Soul Memory」
    ↓
读 soul-memory/_tree.md（结构）
    ↓
按需读 trunk/004-睡眠周期.md（详情）
```

### Token 效率
- 旧方案：固定加载 2 天 memory（~3K tokens）
- 新方案：按需加载树形索引（~500 tokens）+ 具体内容
