# 003-old-子 Agent 方案.md

## 原始方案

**目标**：通过子 Agent 隔离降低 token 成本

### 设计

1. **Agent 分级**
   - L1: 轻量级 Agent（4B 模型）- 简单问答
   - L2: 标准 Agent（32B 模型）- 一般任务
   - L3: 重型 Agent（大模型）- 复杂推理

2. **Session 路由**
   - 根据任务复杂度自动分配 Agent
   - 用户可手动指定 Agent 级别

3. **成本对比**
   - 预期降低 30-50% token 消耗

## 废弃评估

### 评估日期
2026-03-23

### 废弃原因

**复杂度分析**：
1. 需要维护多个 Agent 配置
2. Session 路由逻辑复杂
3. 用户体验割裂（用户需要理解 Agent 分级）
4. 开发和维护成本 > 节省的 token 成本

**成本对比**：
| 方案 | 开发成本 | 维护成本 | 用户成本 | token 节省 |
|------|---------|---------|---------|-----------|
| 子 Agent 隔离 | 高 | 高 | 中 | 30-50% |
| Soul Memory | 中 | 低 | 低 | 未知但潜力更大 |

**决策**：废弃子 Agent 方案，转向 Soul Memory。

## 教训

- **简单机制优于复杂架构**
- 不要为了优化而增加系统复杂度
- 有时候根本性的重新设计比修补更有效

## 复活条件

Soul Memory 成功后，如果需要精确的 KPI 计量，可以复活本方案的「计量」部分（去掉「隔离」部分）。
